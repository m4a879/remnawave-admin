# Настройка Webhook для уведомлений об изменении подписки

Эта инструкция поможет настроить автоматические уведомления в Telegram при изменении подписки пользователя через бота Бедолага или другие источники.

## Как это работает

1. **Панель Remnawave** отправляет webhook при изменении пользователя
2. **Админский бот** принимает webhook и отправляет уведомление в Telegram топик
3. В уведомлении отображаются все изменения подписки (дата истечения, количество устройств, лимит трафика и т.д.)

## Настройка

### 1. Настройка админского бота

Добавьте в `.env` файл следующие переменные:

```env
# Порт для webhook сервера (по умолчанию 8080)
WEBHOOK_PORT=8080

# Секретный ключ для проверки подписи webhook (рекомендуется минимум 32 символа)
# Сгенерируйте ключ командой: openssl rand -hex 64
WEBHOOK_SECRET=ваш_секретный_ключ_минимум_32_символа
```

### 2. Настройка панели Remnawave

В настройках панели Remnawave (переменные окружения) укажите:

```env
# Включить webhook уведомления
WEBHOOK_ENABLED=true

# URL админского бота для отправки webhook

# Вариант 1: Бот и панель в одной Docker сети (локально)
# ⚠️ ВАЖНО: Используйте HTTP, а не HTTPS для внутренних Docker сервисов!
# Webhook сервер бота работает по HTTP, а не HTTPS
WEBHOOK_URL=http://remnawave-admin-bot:8080/webhook
# или если сервис называется по-другому (проверьте имя в docker-compose.yml):
# WEBHOOK_URL=http://bot:8080/webhook

# Вариант 2: Бот на другом сервере через HTTP (если безопасность позволяет)
# Используйте IP адрес или домен без SSL
WEBHOOK_URL=http://IP-АДРЕС-СЕРВЕРА:8080/webhook
# или
WEBHOOK_URL=http://домен-бота.com:8080/webhook

# Вариант 3: Бот на другом сервере через HTTPS (РЕКОМЕНДУЕТСЯ для продакшена)
# ⚠️ ВАЖНО: Убедитесь, что SSL сертификат валидный и не истек!
# Нужен reverse proxy (nginx, traefik, caddy) с валидным SSL сертификатом
WEBHOOK_URL=https://домен-бота.com/webhook
# или если используется нестандартный порт:
# WEBHOOK_URL=https://домен-бота.com:8443/webhook

# Секретный ключ (должен совпадать с WEBHOOK_SECRET в админском боте)
# Панель будет использовать этот ключ для создания HMAC-SHA256 подписи от тела запроса
# Подпись отправляется в заголовке X-Remnawave-Signature
WEBHOOK_SECRET_HEADER=ваш_секретный_ключ_минимум_32_символа
```

**Важно:** 
- `WEBHOOK_SECRET_HEADER` в панели должен совпадать с `WEBHOOK_SECRET` в админском боте
- Панель Remnawave использует HMAC-SHA256 для создания подписи от тела запроса
- Админский бот автоматически проверяет как HMAC подпись, так и простое сравнение строк (для обратной совместимости)
- События `node.connection_lost` и `node.connection_restored` отправляются личным сообщением всем Telegram ID из `ADMINS`; каждый администратор должен предварительно запустить бота через `/start`

### 3. Настройка Docker Compose

Если используете Docker Compose, убедитесь, что порт проброшен:

```yaml
services:
  bot:
    ports:
      - "8080:8080"  # Порт webhook сервера
    environment:
      WEBHOOK_PORT: 8080
      WEBHOOK_SECRET: ваш_секретный_ключ
```

### 4. Проверка работы

После настройки:

1. Перезапустите админский бот и панель Remnawave
2. Проверьте логи бота - должно появиться сообщение:
   ```
   🌐 Webhook server will be started on port 8080
   ```
3. Проверьте доступность webhook endpoint:
   ```bash
   curl http://localhost:8080/webhook/health
   ```
   Должен вернуться: `{"status":"ok","service":"webhook"}`

4. Сделайте тестовое изменение подписки пользователя через бота Бедолага
5. Проверьте, что уведомление пришло в Telegram топик

## Поддерживаемые события

Админский бот обрабатывает все события из официального списка Remnawave:

### События пользователей (`user.*`)
- `user.created` - создание пользователя
- `user.modified` - изменение пользователя (включая подписку)
- `user.deleted` - удаление пользователя
- `user.revoked` - отзыв подписки
- `user.disabled` - отключение пользователя
- `user.enabled` - включение пользователя
- `user.limited` - ограничение пользователя
- `user.expired` - истечение подписки
- `user.traffic_reset` - сброс трафика
- `user.expires_in_72_hours` - подписка истекает через 72 часа
- `user.expires_in_48_hours` - подписка истекает через 48 часов
- `user.expires_in_24_hours` - подписка истекает через 24 часа
- `user.expired_24_hours_ago` - подписка истекла 24 часа назад
- `user.first_connected` - первое подключение пользователя
- `user.bandwidth_usage_threshold_reached` - достигнут порог использования трафика
- `user.not_connected` - пользователь не подключался длительное время

### События HWID устройств (`user_hwid_devices.*`)
- `user_hwid_devices.added` - добавлено HWID устройство
- `user_hwid_devices.deleted` - удалено HWID устройство

### События нод (`node.*`)
- `node.created` - создана нода
- `node.modified` - изменена нода
- `node.deleted` - удалена нода
- `node.disabled` - нода отключена
- `node.enabled` - нода включена
- `node.connection_lost` - потеряно соединение с нодой
- `node.connection_restored` - соединение с нодой восстановлено
- `node.traffic_notify` - уведомление о трафике ноды

### События сервиса (`service.*`)
- `service.panel_started` - панель запущена
- `service.login_attempt_failed` - неудачная попытка входа
- `service.login_attempt_success` - успешный вход

### События ошибок (`errors.*`)
- `errors.bandwidth_usage_threshold_reached_max_notifications` - достигнут максимум уведомлений о трафике

### События CRM (`crm.*`)
- `crm.infra_billing_node_payment_in_7_days` - оплата ноды через 7 дней
- `crm.infra_billing_node_payment_in_48hrs` - оплата ноды через 48 часов
- `crm.infra_billing_node_payment_in_24hrs` - оплата ноды через 24 часа
- `crm.infra_billing_node_payment_due_today` - оплата ноды сегодня
- `crm.infra_billing_node_payment_overdue_24hrs` - оплата ноды просрочена на 24 часа
- `crm.infra_billing_node_payment_overdue_48hrs` - оплата ноды просрочена на 48 часов
- `crm.infra_billing_node_payment_overdue_7_days` - оплата ноды просрочена на 7 дней

## Формат уведомлений

### Уведомления о пользователях
Содержат полную информацию о пользователе:
- ✅ Username пользователя
- 📶 Лимит трафика (старое → новое значение)
- ⏳ Дата истечения подписки (старое → новое значение)
- 💻 HWID лимит устройств (старое → новое значение)
- 🔗 Ссылка на подписку
- 👥 Внутренний сквад
- И другие параметры, если они изменились

### Уведомления о нодах
Содержат информацию о ноде:
- 🖥 Название ноды
- 🆔 UUID ноды
- 🌐 Адрес и порт
- 🌍 Страна
- 📊 Статус
- 📶 Лимит трафика (если есть)

### Уведомления о сервисе
Содержат информацию о событиях панели:
- 👤 Username (для событий входа)
- 🌐 IP адрес
- 🔍 User Agent

### Уведомления о HWID устройствах
Содержат информацию о добавлении/удалении устройств:
- 👤 Пользователь
- 💻 HWID устройства
- 📅 Дата создания

### Уведомления CRM (биллинг)
Содержат информацию о платежах за инфраструктуру:
- 🖥 Нода
- 🏢 Провайдер
- 💵 Сумма
- 📅 Дата следующей оплаты

## Безопасность

⚠️ **Важно:** Обязательно настройте `WEBHOOK_SECRET`! Без него любой может отправлять webhook запросы на ваш бот.

Рекомендуется:
- Использовать длинный случайный ключ (минимум 32 символа)
- Генерировать ключ командой: `openssl rand -hex 64`
- Не хранить ключ в открытом виде в репозитории

## Устранение проблем

### Webhook не работает

1. Проверьте, что `WEBHOOK_ENABLED=true` в панели
2. Проверьте, что `WEBHOOK_URL` указывает на правильный адрес
3. Проверьте, что порт проброшен в Docker
4. Проверьте логи панели на наличие ошибок отправки webhook
5. Проверьте логи админского бота

### Ошибка SSL/TLS (EPROTO, tlsv1 alert internal error)

Если в логах панели видите ошибку:
```
Error: Failed to send webhook after 3 retries: write EPROTO ...:error:0A000438:SSL routines:ssl3_read_bytes:tlsv1 alert internal error
```

**Причины и решения:**

1. **Использование HTTPS для внутренних Docker сервисов**
   - ❌ **Неправильно:** `WEBHOOK_URL=https://remnawave-admin-bot:8080/webhook`
   - ✅ **Правильно:** `WEBHOOK_URL=http://remnawave-admin-bot:8080/webhook`
   - Для сервисов в одной Docker сети всегда используйте HTTP, а не HTTPS

2. **Проблемы с SSL сертификатом на внешнем домене (бот на другом сервере)**
   
   ⚠️ **ВАЖНО:** Webhook сервер бота работает **ТОЛЬКО по HTTP**, а не HTTPS!
   
   **Если вы указали HTTPS URL в панели, но не настроили reverse proxy:**
   - ❌ **Неправильно:** `WEBHOOK_URL=https://IP-АДРЕС:8080/webhook` (без reverse proxy)
   - ✅ **Правильно:** `WEBHOOK_URL=http://IP-АДРЕС:8080/webhook` (используйте HTTP)
   
   **Варианты решения:**
   
   **Вариант A (проще):** Используйте HTTP для внешнего сервера:
   ```
   WEBHOOK_URL=http://IP-АДРЕС-СЕРВЕРА:8080/webhook
   # или если есть домен:
   WEBHOOK_URL=http://домен-бота.com:8080/webhook
   ```
   ⚠️ Убедитесь, что порт 8080 доступен с сервера панели (firewall, security groups)
   
   **Вариант B (безопаснее):** Настройте reverse proxy с HTTPS:
   - Webhook сервер бота работает по HTTP на порту 8080 (внутри сервера)
   - Reverse proxy (nginx, traefik, caddy) принимает HTTPS и проксирует на `http://localhost:8080/webhook`
   - Используйте Let's Encrypt для бесплатного SSL сертификата
   - Затем используйте: `WEBHOOK_URL=https://домен-бота.com/webhook`
   
   **Пример конфигурации nginx для reverse proxy:**
   ```nginx
   server {
       listen 443 ssl http2;
       server_name ваш-домен.com;
       
       # SSL сертификат (Let's Encrypt)
       ssl_certificate /etc/letsencrypt/live/ваш-домен.com/fullchain.pem;
       ssl_certificate_key /etc/letsencrypt/live/ваш-домен.com/privkey.pem;
       
       location /webhook {
           proxy_pass http://localhost:8080;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
           proxy_set_header X-Forwarded-Proto $scheme;
       }
   }
   ```
   
   **Пример конфигурации Caddy (Caddyfile):**
   ```caddy
   ваш-домен.com {
       # Вариант 1: Если Caddy и бот в одной Docker сети
       reverse_proxy /webhook bot:8080
       
       # Вариант 2: Если Caddy на хосте, а бот в Docker (порт проброшен)
       # Используйте 127.0.0.1 вместо localhost, чтобы избежать проблем с IPv6
       # reverse_proxy /webhook 127.0.0.1:8080
       
       # Вариант 3: Если используете имя сервиса из docker-compose.yml
       # reverse_proxy /webhook remnawave-admin-bot:8080
   }
   ```
   
   ⚠️ **Важно для Caddy:**
   - **Если Caddy и бот в одной Docker сети:** используйте имя сервиса `bot:8080` (из docker-compose.yml)
   - **Если Caddy на хосте, а бот в Docker:** используйте `127.0.0.1:8080` (НЕ `localhost:8080`, чтобы избежать проблем с IPv6)
   - **Если получаете ошибку `connection refused`:** проверьте, что порт 8080 проброшен на хост в docker-compose.yml
   - **Если получаете ошибку `no such host`:** убедитесь, что Caddy и бот в одной Docker сети, или используйте IP адрес
   
   **Устранение ошибки "connection refused":**
   - Убедитесь, что бот слушает на `0.0.0.0:8080` (не на `127.0.0.1:8080`)
   - Проверьте, что порт проброшен: `docker-compose.yml` должен содержать `ports: - "8080:8080"`
   - Если Caddy в Docker, используйте имя контейнера/сервиса вместо `localhost`
   - Используйте `127.0.0.1` вместо `localhost` для принудительного использования IPv4

3. **Проверка доступности webhook endpoint**
   ```bash
   # Из контейнера панели проверьте доступность:
   curl http://remnawave-admin-bot:8080/webhook/health
   # Должен вернуться: {"status":"ok","service":"webhook"}
   ```

4. **Проверка имени сервиса в Docker**
   - Убедитесь, что имя сервиса в `WEBHOOK_URL` совпадает с именем сервиса в `docker-compose.yml`
   - Если сервис называется `bot`, используйте: `http://bot:8080/webhook`
   - Если сервис называется `remnawave-admin-bot`, используйте: `http://remnawave-admin-bot:8080/webhook`

### Уведомления не приходят

1. Убедитесь, что `NOTIFICATIONS_CHAT_ID` настроен в админском боте
2. Проверьте, что бот имеет доступ к указанному чату/топику
3. Проверьте логи админского бота на наличие ошибок

### Ошибка 401 Unauthorized

Ошибка `Failed to send webhook after 3 retries: Request failed with status code 401` означает, что бот отклоняет запрос из-за несоответствия секретного ключа.

**Как работает проверка подписи:**

Панель Remnawave использует **HMAC-SHA256** для создания подписи от тела запроса. Админский бот автоматически:
1. Вычисляет HMAC-SHA256 от тела запроса с использованием `WEBHOOK_SECRET`
2. Сравнивает вычисленную подпись с подписью из заголовка `X-Remnawave-Signature`
3. Также поддерживает простое сравнение строк (для обратной совместимости)

**Причины и решения:**

1. **Секретные ключи не совпадают**
   - Убедитесь, что `WEBHOOK_SECRET` в админском боте **точно совпадает** с `WEBHOOK_SECRET_HEADER` в панели Remnawave
   - Проверьте, что нет лишних пробелов или символов в значениях
   - Убедитесь, что используются одинаковые переменные окружения (без кавычек, если они не нужны)
   - **Важно:** Панель использует этот ключ для создания HMAC подписи, поэтому ключи должны быть идентичными

2. **Заголовок не отправляется**
   - Проверьте логи админского бота - должно быть предупреждение: `X-Remnawave-Signature header missing`
   - Убедитесь, что в панели Remnawave установлена переменная `WEBHOOK_SECRET_HEADER`
   - Перезапустите панель Remnawave после изменения переменных окружения

3. **Проблемы с кодировкой или форматом**
   - Убедитесь, что секретный ключ не содержит специальных символов, которые могут быть неправильно интерпретированы
   - Рекомендуется использовать только буквы, цифры и дефисы
   - Проверьте логи бота - там будет указана длина ожидаемого и полученного ключа

4. **Проверка через логи**
   - В логах админского бота теперь будет подробная информация:
     - Какие заголовки были получены
     - Длина ожидаемого и полученного ключа
     - Первые символы полученного ключа (для сравнения без раскрытия полного значения)
   - Ищите сообщения с `Webhook signature mismatch` или `X-Remnawave-Signature header missing`

**Быстрая проверка:**
```bash
# Проверьте, что секреты совпадают (замените на ваши значения)
echo "В боте: $WEBHOOK_SECRET"
echo "В панели: $WEBHOOK_SECRET_HEADER"
# Они должны быть идентичны
```

## Тестирование

Для тестирования webhook можно использовать curl:

```bash
# Тест health endpoint
curl http://localhost:8080/webhook/health

# Тест отправки уведомления (требует userUuid)
curl -X POST http://localhost:8080/webhook/test \
  -H "Content-Type: application/json" \
  -H "X-Remnawave-Signature: ваш_секретный_ключ" \
  -d '{"userUuid": "uuid-пользователя"}'
```
